# 大田精准耕播智能决策系统 - 技术架构文档

> 本文档基于"DSL必然要求可审计的软件系统设计"原则，核心是**裁决层集中管理** + **可审计性** + **证据链推理**

> **宣讲文档**：[《传统Agent vs 大田精准耕播智能决策系统 — 推理链路与运行时对比》](./《传统Agent%20vs%20大田精准耕播智能决策系统%20—%20推理链路与运行时对比》.md) — 一句话讲清传统 Agent 和本项目在推理链路与运行时上的本质差异

---

## 一、核心设计原则

| 原则 | 说明 |
|------|------|
| **裁决层集中** | 所有业务规则集中在裁决层，禁止分散到组件内部 |
| **能力对象消费** | 组件只消费能力对象，不拼接业务规则 |
| **证据链推理** | 推理过程必须包含证据收集、评估、路径推导、置信度 |
| **可审计性** | 所有规则变更可追溯、可测试 |
| **状态替代条件** | 用状态代替条件判断，避免 if 散落 |

## 二、系统分层 + Agent 框架架构

### 2.1 Agent 框架选型说明

| 框架 | 适用场景 | 是否采用 | 原因 |
|------|----------|----------|------|
| **ReAct** | 推理 → 行动循环 | ✅ 采用 | LangGraph 原生支持，状态流转清晰 |
| **Plan-and-Execute** | 先规划后执行 | ❌ 不采用 | 执行中无法调整，不适合敏捷迭代 |
| **Tool-Augmented LLM** | 工具调用 | ✅ 辅助 | 扩展 LLM 能力 |
| **Multi-Agent** | 多 Agent 协作 | ❌ 暂不采用 | 团队分工简单，暂不需要 |
| **Memory-Augmented** | 记忆持久化 | ✅ 采用 | 项目状态持久化、上下文保持 |
| **Autonomous Agent** | 端到端自主 | ❌ 不采用 | 不可控、难以调试 |
| **Tree-of-Thoughts** | 树形探索 | ❌ 不采用 | 推理路径已知，无需探索 |
| **Graph-of-Thoughts** | 图结构推理探索 | ❌ 不采用 | 过度设计，不需要 |

## 三、证据链推理链路设计

### 3.1 核心概念

**不是简单的 RAG → Prompt → Response**，而是需要有完整的**证据链**和**推理路径**：

```
┌─────────────────────────────────────────────────────────────────┐
│                     证据链推理 vs 普通RAG                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  普通 RAG：                                                    │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐                    │
│  │ 检索   │ -> │ 拼接   │ -> │ 输出   │                    │
│  │ 召回   │    │ Prompt │    │ 响应   │                    │
│  └─────────┘    └─────────┘    └─────────┘                    │
│                       ❌ 自由度太大，无法追溯                    │
│                                                                  │
│  证据链推理：                                                   │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐   │
│  │ 证据   │ -> │ 评估   │ -> │ 推理   │ -> │ 置信度  │   │
│  │ 收集   │    │ 可靠性  │    │ 路径   │    │ 输出   │   │
│  └─────────┘    └─────────┘    └─────────┘    └─────────┘   │
│                       ✅ 每步可追溯，结论有据                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 四、证据链类型定义

### 4.1 证据类型

```typescript
// ===== 证据来源类型 =====
type EvidenceSource = 'document' | 'task' | 'economic' | 'history' | 'team_input' | 'sensor'

// ===== 证据结构 =====
interface Evidence {
  id: string
  source: EvidenceSource
  type: string
  content: string                    // 证据内容
  reliability: number                // 可靠性：[0-1]
  relevance: number                  // 关联性：[0-1]
  timestamp: string                  // 证据时间
  expires_at?: string                // 过期时间（可选）
  metadata: {
    source_doc?: string              // 来源文档
    source_id?: string               // 来源ID（任务ID/用户ID等）
    from_user?: string               // 提供者
    confidence?: number              // 原始置信度
    [key: string]: any
  }
}

// ===== 证据链 =====
interface EvidenceChain {
  id: string
  query: string                     // 原始问题
  created_at: string
  evidences: Evidence[]
  reasoning_steps: ReasoningStep[]
  conclusion: Conclusion
  confidence: ConfidenceBreakdown
  traces: string[]                  // 推理痕迹
}

// ===== 推理步骤 =====
interface ReasoningStep {
  step: number
  action: 'evidence_integration' | 'path_deduction' | 'hypothesis_validation' | 'conclusion_generation'
  input_evidence: string[]          // 引用的证据IDs
  intermediate_result: any          // 中间结果
  description: string               // 步骤描述
}

// ===== 结论 =====
interface Conclusion {
  type: 'decision' | 'recommendation' | 'analysis' | 'warning'
  content: string                   // 结论内容
  actions?: string[]                // 建议行动
  risks?: RiskItem[]                // 风险提示
}

// ===== 置信度分解 =====
interface ConfidenceBreakdown {
  base_confidence: number            // 基础置信度
  reliability_discount: number      // 证据可靠性折扣
  conflict_discount: number         // 冲突/风险折扣
  final_confidence: number          // 最终置信度
}

// ===== 风险项 =====
interface RiskItem {
  level: 'low' | 'medium' | 'high'
  description: string
  probability: number               // 发生概率 [0-1]
  impact: string                    // 影响描述
}
```

```typescript
- 最终置信度：X%
```

```typescript
    hasAudio: false,
    textContent: lastMessage.content,

      intent: parsed.intent,
```

```typescript
// src/agents/edges/conditional.ts

import { AgentState } from '../state'
import { agentAuditRoute } from '@/lib/agent-audit-logger'

// ===== 意图路由：所有意图都走 retrieval（RAG 始终参与，禁止 AI 随意发挥）=====
export function routeByIntent(state: typeof AgentState.State) {
  const intent = state.currentTask?.intent || state.explicitIntent

  if (!intent) {
    agentAuditRoute('intention', 'retrieval', '无意图, 默认走检索')
    return 'retrieval'
  }

  agentAuditRoute('intention', 'retrieval', `intent=${intent} (所有意图都走RAG)`)
  return 'retrieval'
}

// ===== 交互点路由：有交互点→response（含询问），无交互点→verdict =====
export function routeByInteractionPoint(state: typeof AgentState.State) {
  const { pendingInteraction } = state

  if (pendingInteraction) {
    agentAuditRoute('interactionPointDetection', 'response', `有交互点: type=${pendingInteraction.type}`)
    return 'response'
  }

  agentAuditRoute('interactionPointDetection', 'verdict', '无交互点, 走裁决')
  return 'verdict'
}

// ===== 裁决路由：当前所有裁决类型都走 response =====
export function routeByVerdictType(state: typeof AgentState.State) {
  const verdictResult = state.verdictResult

  if (!verdictResult) {
    agentAuditRoute('verdict', 'response', '无裁决结果')
    return 'response'
  }

  agentAuditRoute('verdict', 'response', `verdictType=${verdictResult.type}`)
  return 'response'
}
```

### 7.5 Agent 编译

```typescript
// src/agents/index.ts

import { StateGraph } from '@langchain/langgraph'
import { AgentState } from './state'
import { intentionNode, retrievalNode, reasoningNode, interactionPointDetectionNode, verdictNode, responseNode } from './nodes'
import { routeByIntent, routeByInteractionPoint, routeByVerdictType } from './edges/conditional'
import { MemorySaver } from '@langchain/langgraph'

// ===== 创建 Agent =====

const workflow = new StateGraph(AgentState)
  .addNode('intention', async (state) => wrapNodeWithAudit('intention', state, intentionNode))
  .addNode('retrieval', async (state) => wrapNodeWithAudit('retrieval', state, retrievalNode))
  .addNode('reasoning', async (state) => wrapNodeWithAudit('reasoning', state, reasoningNode))
  .addNode('interactionPointDetection', async (state) => wrapNodeWithAudit('interactionPointDetection', state, interactionPointDetectionNode))
  .addNode('verdict', async (state) => wrapNodeWithAudit('verdict', state, verdictNode))
  .addNode('response', async (state) => wrapNodeWithAudit('response', state, responseNode))
  .addEdge('__start__', 'intention')
  .addConditionalEdges('intention', routeByIntent, ['retrieval'])
  .addEdge('retrieval', 'reasoning')
  .addEdge('reasoning', 'interactionPointDetection')
  .addConditionalEdges('interactionPointDetection', routeByInteractionPoint, ['verdict', 'response'])
  .addConditionalEdges('verdict', routeByVerdictType, ['response'])
  .addEdge('response', '__end__')

// ===== 编译并添加持久化 =====

const checkpointer = new MemorySaver()

export const agent = workflow.compile({
  checkpointer,
  debug: process.env.NODE_ENV === 'development',
})

// ===== Agent 图结构 =====
//
// __start__ → intention → (routeByIntent: 所有意图→retrieval)
//   → retrieval → reasoning → interactionPointDetection → (routeByInteractionPoint)
//       ├── 有交互点 → response → __end__
//       └── 无交互点 → verdict → (routeByVerdictType: 所有→response) → __end__
//
// 关键设计决策：
// 1. 所有意图都走 retrieval — RAG 始终参与，禁止 AI 随意发挥
// 2. 交互点检测 — 仅 analysis/planning 意图下激活
// 3. 每个节点都包装审计 — wrapNodeWithAudit 记录开始/结束/错误日志
// 4. 链路追踪 — startChainTrace/endChainTrace 记录完整执行链路
```

### 7.7 流式事件总线架构

#### 7.7.1 问题背景

LangGraph `agent.stream()` 的粒度是**每个节点一个 chunk**。节点内部有丰富数据（证据、推理步骤等），但节点边界 `async (state) => result` 导致这些数据无法实时外泄到 SSE。response 节点使用 `invoke()` 全量生成后再拆分为 3 字符假流式，所有 token 在同一毫秒推送，GUI 表现为"等待 60 秒 → 全部内容一次性出现"。

#### 7.7.2 事件总线机制

新增 `src/agents/stream-bus.ts`，模块级事件注册表，绕过 LangGraph 节点边界：

```typescript
// 模块级 Map，key = traceId，value = SSE 推送回调
const streamCallbacks = new Map<string, (event: StreamEvent) => void>()

// route.ts 在启动 agent 前注册
registerStreamBus(traceId, (event) => {
  controller.enqueue(encoder.encode(`data: ${JSON.stringify(event)}\n\n`))
})

// 各节点内直接调用，同步推送到 SSE（不依赖节点 return）
emitStreamEvent(traceId, { type: "evidence_found", evidence: {...} })
```

#### 7.7.3 三层事件架构

```
┌─────────────────────────────────────────────────────────────────┐
│ Layer A: 运行时流式（节点执行过程中实时推送）                      │
├─────────────────────────────────────────────────────────────────┤
│ retrieval 节点:                                                  │
│   rag_search      → ChromaDB 查询前推送                          │
│   evidence_found  → 循环内逐条推送（每条证据到达即推送）           │
│   rag_result      → 查询完成后推送统计                            │
│                                                                  │
│ response 节点:                                                   │
│   response_start  → LLM.stream() 开始前推送                      │
│   token           → LLM.stream() 逐 token 推送（真流式）          │
│   response_end    → 流式结束后推送完整文本                        │
├─────────────────────────────────────────────────────────────────┤
│ Layer B: 节点结束时流式（节点 return 后推送结构化增量）             │
├─────────────────────────────────────────────────────────────────┤
│ intention 结束:     structured_update { intent }                 │
│ retrieval 结束:     structured_update { evidenceChain }          │
│ reasoning 结束:     structured_update { reasoningPath }          │
│ interactionPointDetection 结束: structured_update { interactionPoint } │
│ verdict 结束:       structured_update { verdict }                │
│ response 结束:      structured_update { displayContent }         │
├─────────────────────────────────────────────────────────────────┤
│ Layer C: 完成                                                    │
├─────────────────────────────────────────────────────────────────┤
│ done  { threadId, traceId, chainTrace, worldLines }              │
└─────────────────────────────────────────────────────────────────┘
```

#### 7.7.4 各节点推送时序

```
route.ts:

  agentStream = await streamAgent(...)

  ┌─ intention ────────────────────────────────────┐
  │  invoke(prompt) → parse →                      │

  │  }                                             │

  │  emitStreamEvent("structured_update",          │
  │    { evidenceChain })                          │
  │  return { evidenceChain }                      │
  └────────────────────────────────────────────────┘

  ┌─ reasoning ────────────────────────────────────┐
  │  invoke(prompt) → parse →                      │
  │  emitStreamEvent("structured_update",          │
  │    { reasoningPath: { steps, traces } })       │
  │  return { verdictResult }                      │

  ┌─ verdict ──────────────────────────────────────┐
```

## 八、farm-agent 核心架构演进（第1-3轮）

> farm-agent 相比 team-coordinator-agent 功能范围有所精简（不含 worldline/L0-L4/VerdictRegistry/CapabilityModel），保留核心六节点决策管线。代码实现质量要求与 team-coordinator-agent 完全一致，禁止任何形式的简化实现。
>
> 以下为 farm-agent 在类型收敛、响应裁决、跨轮记忆三个维度的架构演进。

### 8.1 第1轮：类型收敛 + thinkingLevel 路由

#### 8.1.1 EvidenceSource 统一

所有 evidence 来源收敛到唯一类型 `EvidenceSource`，覆盖知识库检索和父 Plan 预留场景：

```typescript
type EvidenceSource =
  | "document"       // 知识库文档片段
  | "knowledge"      // 知识库检索结果
  | "knowledge_empty" // 知识库无结果
  | "project_context" // 项目上下文
  | "keywords"       // 关键词扩展
  | "multimodal"     // 多模态输入
  | "task"           // 任务数据
  | "economic"       // 经济数据
  | "history"        // 历史对话
  | "team_input"     // 团队输入
  | "sensor"         // 传感器数据
```

每一条 evidence 增加 `chunkId` 追溯字段，只能从真实 metadata 提取（`metadata.chunkId` / `metadata.vectorId` / `documentId + chunkIndex`），禁止用 random UUID 伪造。

#### 8.1.2 thinkingLevel 路由

引入 `ThinkingLevel` 类型实现快/深路由分流：

```typescript
type ThinkingLevel = "fast" | "deep"
```

```text
intention 节点
  │ routeByIntent
  ├─ thinkingLevel === "fast" → response（跳过 RAG + 推理）
  └─ thinkingLevel === "deep" → retrieval（完整证据链推理）
```

- **fast**：明确可跳过深链路时直接回复（问候、闲聊等），输出仅 conclusion section
- **deep**：走完整 RAG + reasoning + verdict 链路
- **fallback**：缺失 thinkingLevel 时保守走 deep，防止跳过必要推理

### 8.2 第2轮：ResponseStrategy 响应裁决

将 response 节点中分散的 prompt 约束、sections 组合和回复策略抽象为独立的 StrategyDescriptor 注册表：

```
ResponseStrategy 集中裁决
  │
  ├─ StrategyDescriptor registry（8 个 descriptor，自声明 matches）
  │   ├─ fast:chat          — 问候/闲聊，sections 仅 conclusion
  │   ├─ deep:analysis      — 分析类，sections 含 evidence + reasoning + confidence + risk
  │   ├─ deep:planning      — 规划类，sections 含 action_steps + timeline
  │   ├─ deep:decision      — 决策类
  │   ├─ deep:question      — 问答类，sections 含 evidence_digest
  │   ├─ deep:creation      — 创作类
  │   ├─ deep:modification  — 修改类
  │   └─ deep:fallback      — 兜底（priority=1，永远可匹配）
  │
  └─ resolveResponseStrategy(ctx) → ResponseStrategy
       ├─ promptHint      → 注入 LLM prompt
       ├─ sections        → 决定 DisplayContent 输出哪些 section
       ├─ maxTokens       → 长度控制元数据
       └─ showEvidenceDigest → evidence_digest 开关
```

**核心原则**：
- response 节点不再保留策略硬编码三段分支，只消费裁决结果
- `DisplayContent.sections` 统一使用单数形式（`"risk"` 而非 `"risks"`）
- `evidence_digest` 依赖 `hasNonPriorEvidence` 判定：仅当存在知识库文档（含 chunkId/documentName）或运行时业务数据源时才展示

### 8.5 第5轮：语义缓存 — 避免重复管线执行

> **子文档**：[`.ai/specs/co-agent-semantic-cache/spec.md`](../../../.ai/specs/co-agent-semantic-cache/spec.md) — 语义缓存详细 spec（三元组缓存键 + 三层淘汰 + 模拟流式输出）

#### 8.5.1 问题背景

相同问题重复发送给 Agent 时每次都走完整 6 节点管线，浪费 LLM token 和响应延迟。需要一层**语义级缓存**：相同意图 + 相同问题 + 相同激活专家 → 直接返回缓存结果。

传统缓存方案在 Agent 管线中面临两个挑战：
- **缓存键构建需要 intent**（intention 节点产出），不能在管线启动前查询
- **知识库增量更新**后旧缓存需要失效，但不能全局清空（太粗暴）

#### 8.5.2 架构位置

```
POST /api/agent/chat/stream
  │
  ├─ intention 节点（产出 intent + thinkingLevel + activeExperts）
  │   │
  │   ├─ thinkingLevel=fast → response（跳过缓存，fast 不涉及 RAG）
  │   │
  │   └─ thinkingLevel=deep → 构建 cacheKey → 查询 SemanticCache
  │       │
  │       ├─ HIT → 模拟流式输出 → response（跳过 retrieval+reasoning+verdict）
  │       │
  │       └─ MISS → retrieval → reasoning → ... → response
  │             └─ 管线完成后 → semanticCache.set(cacheKey, entry) (fire-and-forget)
```

关键设计决策：缓存查询插在 `intention` 节点之后、`retrieval` 之前，而非管线启动之前——因为 `intent` 和 `activeExperts` 由 `intention` 节点产出。

#### 8.5.3 缓存键（三元组）

```typescript
interface CacheKey {
  normalizedQueryHash: string     // normalizeQuery(text) 的 SHA256 前 16 位
  intentHash: string              // intent 的 hash 前 8 位
  activeExpertsHash: string       // sorted(activeExpertIds).join(",") 的 hash 前 8 位
  compositeKey: string            // `${normalizedQueryHash}:${intentHash}:${activeExpertsHash}`
}
```

`kbGeneration` 不在 key 中，而是作为 `CacheEntry` 的出生版本号字段，在 `get()` 中通过字段比对执行惰性淘汰。

#### 8.5.4 三层淘汰策略

`get(key)` 中按以下顺序判定，职责边界清晰：

| 层次 | 判定条件 | 语义 |
|------|---------|------|
| **Layer 1 — Generation** | `entry.kbGeneration < getKbGeneration()` | 知识库更新后旧缓存失效 |
| **Layer 2 — TTL** | `Date.now() - entry.createdAt > ttl * 1000` | 时间自然过期 |
| **Layer 3 — LRU** | `size > MAX_CACHE_SIZE(200)` | 内存容量保护 |

```typescript
interface CacheEntry {
  responseContent: string          // 完整回复文本
  displayContent: Record<string, unknown>  // structured_output
  createdAt: number                // Date.now() 时间戳
  ttl: number                      // 按意图分档（chat=3600, analysis=300, ...）
  hitCount: number                 // 命中次数
  sourceTraceId: string            // 原始 traceId（可追溯）
  kbGeneration: number             // 出生版本号
  intent: string
}
```

TTL 按意图类型分档：

| intent | TTL | 原因 |
|--------|-----|------|
| chat | 3600s (1h) | 闲聊内容敏感度低 |
| question | 1800s (30m) | 问答时效性中等 |
| analysis | 300s (5m) | 分析需反映最新知识 |
| planning | 600s (10m) | 规划基于相对稳定的知识 |
| decision | 600s (10m) | 同 planning |
| creation/modification | 300s (5m) | 创作类时效性敏感 |
| default | 300s (5m) | 保守兜底 |

#### 8.5.5 kbGeneration 生命周期

```
knowledge-indexer 索引完成后
  → bumpKbGeneration()                    # 全局版本号 +1

semanticCache.get(key):
  → entry.kbGeneration < getKbGeneration()
  → 淘汰该条目（惰性，O(1)，在 get() 中判定）
```

**TTL 与 kbGeneration 的关系**：
- TTL：常规刷新（确保缓存不会永久锁死旧答案）
- kbGeneration：知识库变更时立即失效（而非等待 TTL）
- 两者互补：TTL 保证新鲜度，kbGeneration 保证一致性

#### 8.5.6 缓存命中模拟流式输出

缓存命中时不直接返回完整结果，而是模拟正常 SSE 流式体验：

```
cache_hit event (sourceTraceId + cachedAt)
  → token "根" (3 chars / chunk, 自适应延迟)
  → token "据分"
  → token "析，"
  → ...
  → done event (含 displayContent)
```

总耗时控制在 0.5-1.5s，前端体验与正常流式不可区分。

### 8.6 第6轮：演化闭环 — 路径质量 + TTL 自适应 + turnHistory

#### 8.6.1 设计意图

前面各轮建立了规则嵌入（策略注册、分析专家、语义缓存），但所有参数都是静态常量。演化闭环使系统在运行中从自己的审计数据里学习参数——TTL 从常量自主学习、下载格式偏好从用户行为学习、执行质量下降时自动调整策略。

三个独立回路各自运行，互不阻塞主聊天流程：

```
回路一（缓存层）: 缓存过期 → LLM 重跑 → 对比新旧结论 → TTL 自适应
回路二（报告层）: 下载行为记录 → 频次统计 → 偏好覆盖默认格式
回路三（策略层）: turnHistory 多轮轨迹 → 4 维度检测 → StrategyAdjustment
```

#### 8.6.2 回路一：语义缓存 TTL 自主学习

**问题**：第5轮的 `CACHE_TTL` 是静态常量（chat=3600s, question=1800s, …），无法根据实际数据稳定性自适应调整。

**机制**：`cache-ttl-stats.ts` 在缓存过期后重新运行 LLM 时采集新旧结论对比：

```
缓存过期 → LLM 重跑 → 对比新旧结论置信度
  ├─ 相同（±5%）→ TTL ↑ 20%（不超过 maxTtl = defaultTtl × 10）
  └─ 不同       → TTL 不变或 ↓ 20%（不低于 minTtl = 60s）
```

触发条件：同一场景累积 ≥ 3 次过期事件。

数据持久化到 `logs/cache-ttl-stats.json`，重启后恢复，删除文件即回退初始常量。

**`loadStats()` 持久化恢复的四种边界**：

| 场景 | 系统状态 | 行为 | 理由 |
|------|---------|------|------|
| 文件不存在 (ENOENT) | 首次运行，从未触发过 TTL 学习 | 静默返回，使用 `DEFAULT_TTL` | 正常首次运行，无需告警 |
| 文件权限拒绝 (EACCES) | 部署后权限变更，或容器挂载异常 | `console.warn` 告警，使用 `DEFAULT_TTL` | 非预期故障，需可观测但不阻塞 |
| JSON 解析失败 | 文件损坏、进程崩溃中断写入 | `console.warn` 告警，使用 `DEFAULT_TTL` | 数据安全回退，告警提醒排查 |
| 文件正常读取 | 正常运行 | 恢复 `_adaptedTtl` 和 `_stats` | 正常路径 |

> 约束：所有失败路径均不阻塞请求处理，`loadStats()` 在 `stream/route.ts` 的 `try` 块开头以 `.catch(() => {})` 调用，确保文件系统问题不影响聊天主流程。

#### 8.6.4 回路三：多维度执行质量自检

**问题**：系统运行中可能出现置信度持续下降、证据覆盖面不足、用户频繁追问等质量退化，当前无自动感知机制。

**机制**：`path-metrics.ts` 提供 4 个独立的 `MetricDescriptor` 检测器：

| 维度 | 算法 | 样本门槛 | 修正方向 |
|------|------|:--------:|---------|
| 置信度轨迹 | 线性回归 β 斜率 | ≥ 5 轮 | β < -3 → `augment_prompt` |
| 证据覆盖率 | 连续 3 轮递减 + 低于全局均值 × 0.5 | ≥ 3 轮 | `relax_evidence_filter` |
| 追问率 | followUpCount > 0 占比 | ≥ 5 轮 | ≥ 40% → `activate_expert`（建议 pest_risk） |
| 置信度波动率 | 标准差 σ | ≥ 5 轮 | σ > 15% → `augment_prompt`（细化维度） |

#### 8.6.5 ResponseStrategy 集成点

`resolveResponseStrategy` 在修饰器管道之后调用 `assessExecutionQuality()`，根据 `StrategyAdjustment` 信号调整最终 strategy：

```typescript
function resolveResponseStrategy(ctx: StrategyContext): ResponseStrategy {
  // ... 已有修饰器管道 ...
  const baseStrategy = decorateStrategy(descriptor.apply, ctx)

  // 执行度评估（同步计算，try-catch 降级）
  if (ctx.turnHistory && ctx.baselines && ctx.turnHistory.length >= 3) {
    try {
      const adjustment = assessExecutionQuality(
        ctx.turnHistory,
        (ctx.activeExperts ?? []).map(e => e.id),
        ctx.baselines,
      )
      if (adjustment.promptSupplement) {
        baseStrategy.promptHint = baseStrategy.promptHint
          ? `${baseStrategy.promptHint} ${adjustment.promptSupplement}`
          : adjustment.promptSupplement
      }

      if (adjustment.dominantAction === "relax_evidence_filter") {
        baseStrategy.promptHint = (baseStrategy.promptHint ?? "")
          + " 注意：近期证据量持续下降，可适当放宽证据筛选条件。"
      }
    } catch {
      // 降级：执行度评估失败不影响响应
    }
  }

  return baseStrategy
}
```

#### 8.6.9 裁决层直驱 GUI 模式（Semantic-Driven UI）

回路三引入了系统架构的一个重要范式升级：**裁决层产出的信号数据本身就是 GUI 的驱动源**，而非函数式控制指令。

**函数式控制 vs 裁决层直驱**：

| 模式 | 后端输出 | 前端行为 | 扩展成本 |
|------|---------|---------|:--:|
| 函数式控制 | `showConfidenceWidget(72)` | 接收指令，渲染固定组件 | 每个新 UI 元素需前后端同步改 |
| 裁决层直驱 | `{ signals: [{metric:"evidence_coverage", severity:"high", action:"relax_evidence_filter"}, ...] }` | 遍历 signals，按 metric 查渲染规则，动态组装 | 后端新增 metric → 前端 zero-change |

**核心设计**：

- `StrategyAdjustmentEvent` 通过 SSE 流式推送，在 `response_start` 之前到达前端
- 前端 `VerdictDetailPanel` 接收 `signals[]` 数组，逐条渲染，不预知维度数量
- 每个 signal 自带完整渲染元数据：
  - `severity` → 颜色（high=红、medium=黄、low=蓝）
  - `metric` → 标签文本（如 `confidence_trajectory`）
  - `action` → 动作描述（如 `relax_evidence_filter`）
- 信号数据是语义完整的——不依赖前端预定义的模板字段

**与第8轮的关系**：第8轮全局状态模型需要调度多回路协同（回路一 TTL + 回路二格式 + 回路三质量），裁决层直驱模式提供了统一的信号总线。不同回路产出的信号共用同一事件格式和同一 GUI 收敛点（verdict 面板），无需为每个回路单独设计 UI。

1. 后端：`METRICS` 数组追加 descriptor，`AdjustmentAction` 联合类型加 `warn_latency`
2. 前端：**不改代码**——新 signal 自动出现在质量自检列表中，severity 控制颜色，metric 控制标签

```
新维度触发后前端自动渲染：
├─ verdict ─────────────────────
├─ 回路三 质量自检 ────────────
│ 🔵 置信度轨迹 · augment_prompt       ← 已有
│ 🔴 证据覆盖率 · relax_evidence_filter ← 已有
│ 🟡 answer_latency · warn_latency      ← 新增，前端 zero-change
└──────────────────────────────
```

### 6.2 推理引擎实现

```typescript
// ===== 推理引擎实现 =====

class EvidenceChainReasoningEngine implements ReasoningEngine {
  constructor(
    private llm: LLMClient,
    private vectorStore: VectorStore,
    private taskStore: TaskStore,
    private economicStore: EconomicStore
  ) {}

  async reason(query: string, context: Context): Promise<EvidenceChain> {
    // 1. 收集证据
    const evidences = await this.collectEvidences(query, context)

    // 2. 评估证据
    const evaluated = await this.evaluateEvidences(evidences)

    // 3. 多步推理
    const steps = await this.multiStepReasoning(evaluated)

    // 4. 生成结论
    const conclusion = await this.generateConclusion(steps)

    // 5. 计算置信度
    const confidence = this.calculateConfidence(evaluated, steps)

    return {
      id: generateId(),
      query,
      created_at: new Date().toISOString(),
      evidences: evaluated,
      reasoning_steps: steps,
      conclusion: conclusion.result,
      confidence,
      traces: this.generateTraces(steps)
    }
  }

  async collectEvidences(query: string, context: Context): Promise<Evidence[]> {
    const [docEvidences, taskEvidences, economicEvidences] = await Promise.all([
      // 文档证据 - 向量检索
      this.vectorStore.similaritySearch(query, { limit: 5 }),
      // 任务证据 - 当前状态
      this.taskStore.getRelevantTasks(context.projectId),
      // 经济证据 - 相关成本数据
      this.economicStore.getRelevantData(query)
    ])

    return [
      ...this.formatDocEvidences(docEvidences),
      ...this.formatTaskEvidences(taskEvidences),
      ...this.formatEconomicEvidences(economicEvidences)
    ]
  }

  async multiStepReasoning(evaluated: EvaluatedEvidence[]): Promise<ReasoningStep[]> {
    const prompt = this.buildReasoningPrompt(evaluated)

    const response = await this.llm.complete({
      prompt,
      temperature: 0.3,  // 低温度保证推理确定性
      output_format: 'json'
    })

    return this.parseReasoningSteps(response)
  }

  private buildReasoningPrompt(evaluated: EvaluatedEvidence[]): string {
    const evidenceList = evaluated.map(e => `
[E${e.id}] 来源：${e.source}
内容：${e.content}
可靠性：${e.reliability}
关联性：${e.relevance}
`).join('\n\n')

    return EVIDENCE_CHAIN_PROMPT
      .replace('{query}', evaluated[0]?.query || '')
      .replace('{evidence_list}', evidenceList)
  }
}
```

## 七、业务裁决层设计（融合推理链路）

### 7.1 核心理念

**业务裁决层不是简单的"能不能做"，而是深度融合推理链路的"应该怎么做"**

```
┌─────────────────────────────────────────────────────────────────┐
│                   浅层裁决 vs 深度裁决                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  浅层裁决（当前错误设计）：                                      │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐                    │

│  └─────────┘    └─────────┘    └─────────┘                    │
│                       ❌ 只回答能不能做，不回答怎么做            │
│                                                                  │
│  深度裁决（正确设计）：                                          │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐   │
│  │ 用户请求 │ -> │ 证据收集 │ -> │ 推理链路 │ -> │ 决策建议 │   │
│  └─────────┘    └─────────┘    └─────────┘    └─────────┘   │
│                       ✅ 回答做什么、怎么做、风险如何            │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 7.4 裁决器实现示例

```typescript
// ===== 优先级裁决器实现 =====

class PriorityVerdictResolver implements VerdictResolver {
  constructor(
    private llm: LLMClient,
    private evidenceCollector: EvidenceCollector,
    private economicCalculator: EconomicCalculator
  ) {}

  async resolvePriority(query: string, context: Context): Promise<VerdictResult> {
    // 1. 收集多维度证据
    const evidences = await this.collectPriorityEvidences(query, context)

    // 2. 构建裁决 Prompt（融合证据链）
    const prompt = this.buildVerdictPrompt(query, evidences)

    // 3. LLM 推理
    const reasoning = await this.llm.complete({
      prompt,
      temperature: 0.3,
      output_format: 'json'
    })

    // 4. 解析裁决结果
    const verdict = this.parseVerdict(reasoning)

    // 5. 计算置信度
    const confidence = this.calculateConfidence(evidences, verdict)

    return {
      type: 'priority_decision',
      query,
      conclusion: verdict.conclusion,
      evidence_chain: { evidences, ... },
      reasoning_path: verdict.steps,
      confidence,
      traces: verdict.traces
    }
  }

  private async collectPriorityEvidences(
    query: string,
    context: Context
  ): Promise<Evidence[]> {
    const [taskEvidences, economicEvidences, historyEvidences] = await Promise.all([
      // 任务证据：deadline、状态、依赖
      this.collectTaskEvidences(context.projectId),
      // 经济证据：延误成本、资源占用
      this.collectEconomicEvidences(context.projectId),
      // 历史证据：类似任务的完成时间
      this.collectHistoryEvidences(query)
    ])

    return [
      ...taskEvidences,  // E1, E2, E3
      ...economicEvidences,  // E4, E5
      ...historyEvidences  // E6, E7
    ]
  }

  private buildVerdictPrompt(query: string, evidences: Evidence[]): string {
    const evidenceList = evidences.map(e => `
[E${e.id}] ${e.source}: ${e.content}
```

### 7.5 裁决类型定义

```typescript
// ===== 任务优先级裁决 =====

interface PriorityVerdict {
  type: 'priority_decision'
  ranked_tasks: Array<{
    task_id: string
    task_name: string
    priority_score: number
    reasons: string[]
  }>
  recommended_first: string
  estimated_completion: string
}

// ===== 风险评估裁决 =====

interface RiskVerdict {
  type: 'risk_assessment'
  overall_risk_level: 'low' | 'medium' | 'high' | 'critical'
  risk_factors: Array<{
    factor: string
    probability: number
    impact: string
    mitigation: string
  }>
  total_expected_loss?: number
  recommendation: string
}

// ===== 资源分配裁决 =====

interface ResourceVerdict {
  type: 'resource_allocation'
  current_allocation: Record<string, number>
  recommended_allocation: Record<string, number>
  utilization_rate: number
  bottleneck_resources: string[]
  recommendation: string
}

// ===== 成本收益裁决 =====

interface CostBenefitVerdict {
  type: 'cost_benefit'
  investment: number
  expected_return: number
  roi: number
  payback_period: string
  sensitivity_analysis: {
    best_case: number
    worst_case: number
    most_likely: number
  }
  recommendation: 'proceed' | 'caution' | 'defer' | 'reject'
}
```
