# 传统 Agent vs 大田精准耕播智能决策系统 — 推理链路与运行时对比

> 本文档用于宣讲和 team 对齐，阐述传统单阶段 Agent 与本项目六节点决策管线 Agent 在推理链路上的本质差异，以及运行时体验的跃升。

---

## 一、先给结论：一句话版本

| | 传统 Agent | 大田精准耕播智能决策系统 |
|------|-----------|--------------|
| 推理链路 | **一段 Prompt 解决一切** | **六个独立阶段，推理与裁决分离** |
| 运行时 | **"等 30 秒，一次性看到结果"** | **"每一步在干什么都能实时看到"** |

下面展开讲。

---

## 二、推理链路的差异

### 2.1 传统 Agent：一段 Prompt 包揽所有

```
┌──────────────────────────────────────────────────────────────────┐
│                      传统 Agent 推理链路                          │
│                                                                   │
│   用户输入                                                        │
│      │                                                            │
│      ▼                                                            │
│   ┌────────────────────────────────────────────────────────┐    │
│   │              一段巨型 Prompt                             │    │
│   │                                                         │    │
│   │   "你是XX助手，请做以下事情：                            │    │
│   │    1. 理解用户意图                                      │    │
│   │    2. 从知识库检索相关文档                               │    │
│   │    3. 基于文档做推理                                     │    │
│   │    4. 给出最终回答"                                      │    │
│   │                                                         │    │
│   │   → 一次 LLM 调用，输入=Retrieval结果+Prompt             │    │
│   │     LLM 在模型内部自行组织"检索→思考→回答"               │    │
│   └────────────────────────────────────────────────────────┘    │
│      │                                                            │
│      ▼                                                            │
│   最终回复（一段纯文本）                                           │
│                                                                   │
│   问题：                                                          │
│   ❌ 检索了什么？不知道。除非 LLM 主动写出来，否则没有记录。       │
│   ❌ 怎么推理的？不知道。为什么得出这个结论？不知道。              │
│   ❌ 置信度多高？不知道。结论正确吗？无法验证。                    │
│   ❌ 需要人介入吗？无法判断。LLM 自己就决定了。                    │
└──────────────────────────────────────────────────────────────────┘
```

**本质缺陷：把"检索"、"推理"、"裁决"、"回答"四个完全不相关的事情，塞进一次 LLM 调用。** LLM 在自己内部建模这些步骤，但外部完全不可见、不可控、不可验证。

---

### 2.2 大田精准耕播智能决策系统：六阶段分离管线

```
┌──────────────────────────────────────────────────────────────────┐
│                大田精准耕播智能决策系统 推理链路                             │
│                                                                   │
│   用户输入                                                        │
│      │                                                            │
│      ▼                                                            │
│   ┌────────────────────────────────────────────────────────────┐ │
│   │ 阶段1: intention（意图解析）                                 │ │
│   │   ─ 分析用户消息，提取意图类型 (analysis/planning/...)       │ │
│   │   ─ 识别实体 (projectId, taskId, keywords)                   │ │
│   │   ─ 支持显式意图快速通道，跳过 LLM 调用                      │ │
│   │   输出: currentTask { intent, entities, query }              │ │
│   └───────────────────────────┬────────────────────────────────┘ │
│                               │                                    │
│                               ▼                                    │
│   ┌────────────────────────────────────────────────────────────┐ │
│   │ 阶段2: retrieval（证据检索）                                 │ │
│   │   ─ ChromaDB 向量检索（最多 5 条）                           │ │
│   │   ─ 项目上下文注入（reliability=0.9）                        │ │
│   │   ─ 关键词匹配（reliability=0.7）                            │ │
│   │   ─ 多模态标记（图片/音频, reliability=0.6）                 │ │
│   │   输出: evidenceChain[] (每条证据含 source/reliability/...)   │ │
│   └───────────────────────────┬────────────────────────────────┘ │
│                               │                                    │
│                               ▼                                    │
│   ┌────────────────────────────────────────────────────────────┐ │
│   │ 阶段3: reasoning（深度推理）◀── 第一次 LLM 推理调用          │ │
│   │   ─ 证据整合 → 路径推导 → 假设验证 → 结论生成                │ │
│   │   ─ 输出推理步骤序列（reasoning_path）                       │ │
│   │   ─ 计算四元置信度（base - reliability_discount - conflict） │ │
│   │   输出: verdictResult { reasoning_path, conclusion, conf. }  │ │
│   └───────────────────────────┬────────────────────────────────┘ │
│                               │                                    │
│                               ▼                                    │
│   ┌────────────────────────────────────────────────────────────┐ │
│   │ 阶段4: interactionPointDetection（交互点检测）               │ │
│   │   ─ 仅 analysis/planning 激活                               │ │
│   │   ─ 检测四种类型: decision / tradeoff / clarification /     │ │
│   │     confirmation                                           │ │
│   │   ─ 有交互点 → 跳过裁决，先问用户                            │ │
│   │   输出: pendingInteraction | null                           │ │
│   └───────────────────────────┬────────────────────────────────┘ │
│                               │                                    │
│                          ┌────┴────┐                              │
│                    有交互点│         │无交互点                       │
│                    跳裁决  │         │继续裁决                       │
│                          ▼         ▼                              │
│   ┌────────────────────────────────────────────────────────────┐ │
│   │ 阶段5: verdict（业务裁决）◀── 第二次 LLM 推理调用             │ │
│   │   ─ 权重解析                        │ │
│   │   ─ 模型参数动态注入（attentionBias / logitBias / temp）    │ │
│   │   ─ 六种裁决类型: 优先级/风险/资源/成本/时间线/路径          │ │
│   │   ─ 裁决完成后自动清除模型钩子                               │ │
│   │   输出: verdictResult { type, conclusion, confidence }      │ │
│   └───────────────────────────┬────────────────────────────────┘ │
│                               │                                    │
│                               ▼                                    │
│   ┌────────────────────────────────────────────────────────────┐ │
│   │ 阶段6: response（回复生成）◀── 第三次 LLM 调用 (流式)        │ │
│   │   ─ 聚合所有上游数据：intent + evidence + reasoning +       │ │
│   │     interactionPoint + verdict                              │ │
│   │   ─ 三条生成路径：有裁决/有交互点 → 有文档 → 兜底问候       │ │
│   │   ─ getLLM().stream() 真流式逐 token 推送                   │ │
│   │   输出: messages[], structuredResponse (完整结构体)          │ │
│   └────────────────────────────────────────────────────────────┘ │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

**核心差异表：**

| 维度 | 传统 Agent | 大田精准耕播智能决策系统 |
|------|-----------|--------------|
| 推理阶段数 | **1 阶段**（一段 Prompt 全包） | **6 阶段**（每阶段独立职责） |
| LLM 调用次数 | **1 次** | **最多 4 次**（intention/reasoning/interactionPointDetection/response） |
| 检索与推理关系 | 混在一次调用中，不可分离 | retrieval → reasoning 串行，检索结果可审计 |
| 推理与裁决关系 | 混在一起 | reasoning（初步推理）+ **verdict（二次裁决）**，推理与决策分离 |
| 人类介入点 | 无 | interactionPointDetection 自动识别，在裁决前把决策权交还用户 |
| 可审计性 | 一条文本，无法追溯 | 每个阶段独立记录，traceId 全链路追踪 |
| 置信度 | 无（或 LLM 随口说一个数字） | 四元置信度分解（base - reliability - conflict），基于证据计算 |
| 模型参数可控 | 固定参数 | verdict 阶段动态调整 attention/logit/temp |

---

### 2.3 为什么"推理与裁决分离"很重要？

```
传统 Agent:
  Prompt: "你是XX助手，基于这些文档分析项目风险，给出建议。"
  LLM 内部:
    读文档 → 理解风险 → 自己决定什么最重要 → 给出建议
           ↑                           ↑
       这是"推理"                 这是"裁决"（决策）
    混在一次推理中，LLM 自己替你决定了"优先级"和"价值观"

大田精准耕播智能决策系统:
  阶段3 reasoning:  "基于证据，当前项目面临风险 A(概率60%)、风险 B(概率30%)、风险 C(概率10%)"
  阶段5 verdict:    用户调整权重滑块 (经济证据→高权重) →
                    "在优先策略下，风险A每天损失5万，建议优先处理"
             ↑
        用户控制了"什么更重要"——紧迫性 / 成本 / 保守观望
        同一套证据，不同权重 → 不同结论
        用户看到了"为什么得出这个结论"，并且可以改变它
```

## 三、运行时的差异

### 3.1 传统 Agent：黑盒等待 → 一次性输出

```
t=0s  用户发消息 "分析项目风险"
      屏幕显示: "思考中..."
      ┌─────────────────────────────────────────────────────────┐
      │                   什么都看不到                             │
      │                                                          │
      │          检索中？不知道。                                 │
      │          找到几条证据？不知道。                           │
      │          在推理吗？不知道。                               │
      │          要等多久？不知道。                               │
      │                                                          │
      │                      ... 30秒 ...                        │
      └─────────────────────────────────────────────────────────┘
t=30s 一次性弹出全文: "根据分析，项目存在以下风险：1. ... 2. ..."
      ↑
      即使号称"流式输出"，也只是把完整文本按 3 个字符切分后逐字推。
      本质是假流式：LLM 先全量生成，再人工拆分。所有 token 在同一毫秒到达。
```

### 3.3 三层事件推送体系

传统 Agent 只有 "输入 → 输出" 两个端点，中间是黑盒。

大田精准耕播智能决策系统通过 **StreamBus** 实现三层事件推送：

| 层次 | 推送时机 | 事件类型 | 说明 |
|------|---------|---------|------|
| **Layer A** | 节点执行过程中 | `rag_search` / `evidence_found` / `rag_result` / `response_start` / `token` / `response_end` | 绕过 LangGraph 节点边界，在节点 return 之前就推送到 SSE。证据一条一条出，token 一个字一个字推。 |
| **Layer B** | 节点 return 后 | `structured_update` { intent / evidenceChain / reasoningPath / interactionPoint / verdict / displayContent } | 每个节点结束后的结构化增量数据，前端渲染专用面板（证据链面板、推理路径面板、置信度面板等）。 |
| **Layer C** | 管线结束 | `done` { threadId, traceId, chainTrace } | 最终完成信号。 |

### 3.4 运行时差异总结

| 维度 | 传统 Agent | 大田精准耕播智能决策系统 |
|------|-----------|--------------|
| 首屏可见时间 | 等全部完成（~30s） | **2 秒**出意图，**4 秒**出第一条证据 |
| 证据可见性 | 不可见，隐含在 Prompt 中 | 逐条推送，卡片形式实时出现 |
| 推理可见性 | 不可见 | 推理步骤逐条展开，引用证据 ID 可追溯 |
| 流式质量 | 假流式（全量生成后切片） | **真流式**（`LLM.stream()` 逐 token 推送） |
| 结构化输出 | 纯文本 | StructuredAgentResponse 完整结构体（证据/推理/裁决/交互点分离） |
| 交互能力 | 一问一答 | 交互点检测 → 用户决策 → 下一轮管线重新执行 |
| 全链路追踪 | 无 | traceId 贯穿所有节点 + 所有事件，可查询完整调用链 |

## 五、代价与收益

### 5.1 代价

| 代价 | 说明 | 缓解方案 |
|------|------|---------|
| LLM 调用次数增加 | 最多 4 次 vs 传统 1 次 | intention 有显式意图快速通道跳过 LLM；interactionPointDetection 仅两种意图激活 |
| 延迟增加 | 串行阶段累加延迟 | 证据条数限 5 条，prompt 摘要截断 200 字符；流式推送让用户感知延迟更低 |
| 工程复杂度增加 | 六节点 + 三类条件边 + StreamBus + 审计 | 复杂度换来可控性——每一层的复杂度都是可见、可调试、可替换的 |

### 5.2 收益

| 收益 | 说明 |
|------|------|
| **推理可追溯** | 每一步都有结构化输出：用了什么证据、怎么推理、为什么得出这个结论 |
| **决策可控** | 权重策略让用户决定"什么更重要"，而非 LLM 替用户决定 |
| **人机协作** | interactionPointDetection 在裁决前把决策权交还用户 |
| **运行时通透** | 每条证据实时推送，推理步骤逐步展开，用户始终知道"现在在做什么" |
| **全链路审计** | traceId 贯穿所有节点，出问题时查询完整调用链即可定位 |
| **防御性 fallback** | 每个 LLM 调用都有 try-catch + 值域校验，单个节点失败不中断管线 |

## 六、适用场景

| 场景 | 推荐方案 |
|------|---------|
| 闲聊、简单问答 | 传统 Agent 足够（但本项目也走 retrieval，多了证据但不碍事） |
| 信息检索（"XX文档说了什么"） | 大田精准耕播智能决策系统：retrieval 后直接 response，有完整的证据来源 |
| 分析（"项目风险有哪些"） | 大田精准耕播智能决策系统：reasoning → interactionPoint → (可选 verdict) → response，推理过程透明 |
| 决策（"应该优先做哪个任务"） | 大田精准耕播智能决策系统：reasoning（推理出选项）→ interactionPointDetection（检测到 decision 类型 → 先问用户） → verdict（用户指定权重后裁决）→ response |
| 规划（"下季度怎么安排"） | 大田精准耕播智能决策系统：全六节点链路，可探索多种策略 |
