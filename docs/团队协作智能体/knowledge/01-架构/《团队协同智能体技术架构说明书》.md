# 团队协同智能体 - 技术架构文档

> 本文档基于"DSL必然要求可审计的软件系统设计"原则，核心是**裁决层集中管理** + **可审计性** + **证据链推理**

> **宣讲文档**：[《传统Agent vs 团队协同智能体 — 推理链路与运行时对比》](./《传统Agent%20vs%20团队协同智能体%20—%20推理链路与运行时对比》.md) — 一句话讲清传统 Agent 和本项目在推理链路与运行时上的本质差异

---

## 一、核心设计原则

| 原则 | 说明 |
|------|------|
| **裁决层集中** | 所有业务规则集中在裁决层，禁止分散到组件内部 |
| **能力对象消费** | 组件只消费能力对象，不拼接业务规则 |
| **证据链推理** | 推理过程必须包含证据收集、评估、路径推导、置信度 |
| **可审计性** | 所有规则变更可追溯、可测试 |
| **状态替代条件** | 用状态代替条件判断，避免 if 散落 |

## 二、系统分层 + Agent 框架架构

### 2.1 Agent 框架选型说明

| 框架 | 适用场景 | 是否采用 | 原因 |
|------|----------|----------|------|
| **ReAct** | 推理 → 行动循环 | ✅ 采用 | LangGraph 原生支持，状态流转清晰 |
| **Plan-and-Execute** | 先规划后执行 | ❌ 不采用 | 执行中无法调整，不适合敏捷迭代 |
| **Tool-Augmented LLM** | 工具调用 | ✅ 辅助 | 扩展 LLM 能力 |
| **Multi-Agent** | 多 Agent 协作 | ❌ 暂不采用 | 团队分工简单，暂不需要 |
| **Memory-Augmented** | 记忆持久化 | ✅ 采用 | 项目状态持久化、上下文保持 |
| **Autonomous Agent** | 端到端自主 | ❌ 不采用 | 不可控、难以调试 |
| **Tree-of-Thoughts** | 树形探索 | ❌ 不采用 | 推理路径已知，无需探索 |
| **Graph-of-Thoughts** | 图结构推理探索 | ❌ 不采用 | 过度设计，不需要 |

## 三、证据链推理链路设计

### 3.1 核心概念

**不是简单的 RAG → Prompt → Response**，而是需要有完整的**证据链**和**推理路径**：

```
┌─────────────────────────────────────────────────────────────────┐
│                     证据链推理 vs 普通RAG                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  普通 RAG：                                                    │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐                    │
│  │ 检索   │ -> │ 拼接   │ -> │ 输出   │                    │
│  │ 召回   │    │ Prompt │    │ 响应   │                    │
│  └─────────┘    └─────────┘    └─────────┘                    │
│                       ❌ 自由度太大，无法追溯                    │
│                                                                  │
│  证据链推理：                                                   │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐   │
│  │ 证据   │ -> │ 评估   │ -> │ 推理   │ -> │ 置信度  │   │
│  │ 收集   │    │ 可靠性  │    │ 路径   │    │ 输出   │   │
│  └─────────┘    └─────────┘    └─────────┘    └─────────┘   │
│                       ✅ 每步可追溯，结论有据                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 四、证据链类型定义

### 4.1 证据类型

```typescript
// ===== 证据来源类型 =====
type EvidenceSource = 'document' | 'task' | 'economic' | 'history' | 'team_input' | 'sensor'

// ===== 证据结构 =====
interface Evidence {
  id: string
  source: EvidenceSource
  type: string
  content: string                    // 证据内容
  reliability: number                // 可靠性：[0-1]
  relevance: number                  // 关联性：[0-1]
  timestamp: string                  // 证据时间
  expires_at?: string                // 过期时间（可选）
  metadata: {
    source_doc?: string              // 来源文档
    source_id?: string               // 来源ID（任务ID/用户ID等）
    from_user?: string               // 提供者
    confidence?: number              // 原始置信度
    [key: string]: any
  }
}

// ===== 证据链 =====
interface EvidenceChain {
  id: string
  query: string                     // 原始问题
  created_at: string
  evidences: Evidence[]
  reasoning_steps: ReasoningStep[]
  conclusion: Conclusion
  confidence: ConfidenceBreakdown
  traces: string[]                  // 推理痕迹
}

// ===== 推理步骤 =====
interface ReasoningStep {
  step: number
  action: 'evidence_integration' | 'path_deduction' | 'hypothesis_validation' | 'conclusion_generation'
  input_evidence: string[]          // 引用的证据IDs
  intermediate_result: any          // 中间结果
  description: string               // 步骤描述
}

// ===== 结论 =====
interface Conclusion {
  type: 'decision' | 'recommendation' | 'analysis' | 'warning'
  content: string                   // 结论内容
  actions?: string[]                // 建议行动
  risks?: RiskItem[]                // 风险提示
}

// ===== 置信度分解 =====
interface ConfidenceBreakdown {
  base_confidence: number            // 基础置信度
  reliability_discount: number      // 证据可靠性折扣
  conflict_discount: number         // 冲突/风险折扣
  final_confidence: number          // 最终置信度
}

// ===== 风险项 =====
interface RiskItem {
  level: 'low' | 'medium' | 'high'
  description: string
  probability: number               // 发生概率 [0-1]
  impact: string                    // 影响描述
}
```

### 4.2 Prompt 模板

```typescript

- 折扣项：
  - 证据可靠性折扣：-X%
  - 冲突风险折扣：-X%
- 最终置信度：X%

### 推理痕迹
1. 根据[E1]...
2. 根据[E2]...
...
`
```

```typescript
    hasImage: false,
    hasAudio: false,
    textContent: lastMessage.content,
  }

  const prompt = intentionPrompt.build(input)
  const response = await getLLM().invoke(prompt)

  let parsed: IntentionOutput
  try {
    parsed = intentionPrompt.parse(response.content as string)
    if (!intentionPrompt.validate(parsed)) {
      throw new Error('Validation failed')
    }
  } catch {
    parsed = { intent: 'chat', entities: { keywords: [] } }
  }

  return {
    currentTask: {
      ...state.currentTask,
      intent: parsed.intent,
      entities: parsed.entities,

  }
}
```

```typescript
// src/agents/edges/conditional.ts

import { AgentState } from '../state'
import { agentAuditRoute } from '@/lib/agent-audit-logger'

// ===== 意图路由：所有意图都走 retrieval（RAG 始终参与，禁止 AI 随意发挥）=====
export function routeByIntent(state: typeof AgentState.State) {
  const intent = state.currentTask?.intent || state.explicitIntent

  if (!intent) {
    agentAuditRoute('intention', 'retrieval', '无意图, 默认走检索')
    return 'retrieval'
  }

  agentAuditRoute('intention', 'retrieval', `intent=${intent} (所有意图都走RAG)`)
  return 'retrieval'
}

// ===== 交互点路由：有交互点→response（含询问），无交互点→verdict =====
export function routeByInteractionPoint(state: typeof AgentState.State) {
  const { pendingInteraction } = state

  if (pendingInteraction) {
    agentAuditRoute('interactionPointDetection', 'response', `有交互点: type=${pendingInteraction.type}`)
    return 'response'
  }

  agentAuditRoute('interactionPointDetection', 'verdict', '无交互点, 走裁决')
  return 'verdict'
}

// ===== 裁决路由：当前所有裁决类型都走 response =====
export function routeByVerdictType(state: typeof AgentState.State) {
  const verdictResult = state.verdictResult

  if (!verdictResult) {
    agentAuditRoute('verdict', 'response', '无裁决结果')
    return 'response'
  }

  agentAuditRoute('verdict', 'response', `verdictType=${verdictResult.type}`)
  return 'response'
}
```

```typescript
// src/agents/index.ts

import { StateGraph } from '@langchain/langgraph'
import { AgentState } from './state'
import { intentionNode, retrievalNode, reasoningNode, interactionPointDetectionNode, verdictNode, responseNode } from './nodes'
import { routeByIntent, routeByInteractionPoint, routeByVerdictType } from './edges/conditional'
import { MemorySaver } from '@langchain/langgraph'

// ===== 创建 Agent =====

const workflow = new StateGraph(AgentState)
  .addNode('intention', async (state) => wrapNodeWithAudit('intention', state, intentionNode))
  .addNode('retrieval', async (state) => wrapNodeWithAudit('retrieval', state, retrievalNode))
  .addNode('reasoning', async (state) => wrapNodeWithAudit('reasoning', state, reasoningNode))
  .addNode('interactionPointDetection', async (state) => wrapNodeWithAudit('interactionPointDetection', state, interactionPointDetectionNode))
  .addNode('verdict', async (state) => wrapNodeWithAudit('verdict', state, verdictNode))
  .addNode('response', async (state) => wrapNodeWithAudit('response', state, responseNode))
  .addEdge('__start__', 'intention')
  .addConditionalEdges('intention', routeByIntent, ['retrieval'])
  .addEdge('retrieval', 'reasoning')
  .addEdge('reasoning', 'interactionPointDetection')
  .addConditionalEdges('interactionPointDetection', routeByInteractionPoint, ['verdict', 'response'])
  .addConditionalEdges('verdict', routeByVerdictType, ['response'])
  .addEdge('response', '__end__')

// ===== 编译并添加持久化 =====

const checkpointer = new MemorySaver()

export const agent = workflow.compile({
  checkpointer,
  debug: process.env.NODE_ENV === 'development',
})

// ===== Agent 图结构 =====
//
// __start__ → intention → (routeByIntent: 所有意图→retrieval)
//   → retrieval → reasoning → interactionPointDetection → (routeByInteractionPoint)
//       ├── 有交互点 → response → __end__
//       └── 无交互点 → verdict → (routeByVerdictType: 所有→response) → __end__
//
// 关键设计决策：
// 1. 所有意图都走 retrieval — RAG 始终参与，禁止 AI 随意发挥
// 2. 交互点检测 — 仅 analysis/planning 意图下激活
// 3. 每个节点都包装审计 — wrapNodeWithAudit 记录开始/结束/错误日志
// 4. 链路追踪 — startChainTrace/endChainTrace 记录完整执行链路
```

### 7.7 流式事件总线架构

#### 7.7.1 问题背景

LangGraph `agent.stream()` 的粒度是**每个节点一个 chunk**。节点内部有丰富数据（证据、推理步骤等），但节点边界 `async (state) => result` 导致这些数据无法实时外泄到 SSE。response 节点使用 `invoke()` 全量生成后再拆分为 3 字符假流式，所有 token 在同一毫秒推送，GUI 表现为"等待 60 秒 → 全部内容一次性出现"。

#### 7.7.2 事件总线机制

新增 `src/agents/stream-bus.ts`，模块级事件注册表，绕过 LangGraph 节点边界：

```typescript
// 模块级 Map，key = traceId，value = SSE 推送回调
const streamCallbacks = new Map<string, (event: StreamEvent) => void>()

// route.ts 在启动 agent 前注册
registerStreamBus(traceId, (event) => {
  controller.enqueue(encoder.encode(`data: ${JSON.stringify(event)}\n\n`))
})

// 各节点内直接调用，同步推送到 SSE（不依赖节点 return）
emitStreamEvent(traceId, { type: "evidence_found", evidence: {...} })
```

#### 7.7.3 三层事件架构

```
┌─────────────────────────────────────────────────────────────────┐
│ Layer A: 运行时流式（节点执行过程中实时推送）                      │
├─────────────────────────────────────────────────────────────────┤
│ retrieval 节点:                                                  │
│   rag_search      → ChromaDB 查询前推送                          │
│   evidence_found  → 循环内逐条推送（每条证据到达即推送）           │
│   rag_result      → 查询完成后推送统计                            │
│                                                                  │
│ response 节点:                                                   │
│   response_start  → LLM.stream() 开始前推送                      │
│   token           → LLM.stream() 逐 token 推送（真流式）          │
│   response_end    → 流式结束后推送完整文本                        │
├─────────────────────────────────────────────────────────────────┤
│ Layer B: 节点结束时流式（节点 return 后推送结构化增量）             │
├─────────────────────────────────────────────────────────────────┤
│ intention 结束:     structured_update { intent }                 │
│ retrieval 结束:     structured_update { evidenceChain }          │
│ reasoning 结束:     structured_update { reasoningPath }          │
│ interactionPointDetection 结束: structured_update { interactionPoint } │
│ verdict 结束:       structured_update { verdict }                │
│ response 结束:      structured_update { displayContent }         │
├─────────────────────────────────────────────────────────────────┤
│ Layer C: 完成                                                    │
├─────────────────────────────────────────────────────────────────┤
│ done  { threadId, traceId, chainTrace, worldLines }              │
└─────────────────────────────────────────────────────────────────┘
```


### 6.2 推理引擎实现

```typescript
// ===== 推理引擎实现 =====

class EvidenceChainReasoningEngine implements ReasoningEngine {
  constructor(
    private llm: LLMClient,
    private vectorStore: VectorStore,
    private taskStore: TaskStore,
    private economicStore: EconomicStore
  ) {}

  async reason(query: string, context: Context): Promise<EvidenceChain> {
    // 1. 收集证据
    const evidences = await this.collectEvidences(query, context)

    // 2. 评估证据
    const evaluated = await this.evaluateEvidences(evidences)

    // 3. 多步推理
    const steps = await this.multiStepReasoning(evaluated)

    // 4. 生成结论
    const conclusion = await this.generateConclusion(steps)

    // 5. 计算置信度
    const confidence = this.calculateConfidence(evaluated, steps)

    return {
      id: generateId(),
      query,
      created_at: new Date().toISOString(),
      evidences: evaluated,
      reasoning_steps: steps,
      conclusion: conclusion.result,
      confidence,
      traces: this.generateTraces(steps)
    }
  }

  async collectEvidences(query: string, context: Context): Promise<Evidence[]> {
    const [docEvidences, taskEvidences, economicEvidences] = await Promise.all([
      // 文档证据 - 向量检索
      this.vectorStore.similaritySearch(query, { limit: 5 }),
      // 任务证据 - 当前状态
      this.taskStore.getRelevantTasks(context.projectId),
      // 经济证据 - 相关成本数据
      this.economicStore.getRelevantData(query)
    ])

    return [
      ...this.formatDocEvidences(docEvidences),
      ...this.formatTaskEvidences(taskEvidences),
      ...this.formatEconomicEvidences(economicEvidences)
    ]
  }

  async multiStepReasoning(evaluated: EvaluatedEvidence[]): Promise<ReasoningStep[]> {
    const prompt = this.buildReasoningPrompt(evaluated)

    const response = await this.llm.complete({
      prompt,
      temperature: 0.3,  // 低温度保证推理确定性
      output_format: 'json'
    })

    return this.parseReasoningSteps(response)
  }

  private buildReasoningPrompt(evaluated: EvaluatedEvidence[]): string {
    const evidenceList = evaluated.map(e => `
[E${e.id}] 来源：${e.source}
内容：${e.content}
可靠性：${e.reliability}
关联性：${e.relevance}
`).join('\n\n')

    return EVIDENCE_CHAIN_PROMPT
      .replace('{query}', evaluated[0]?.query || '')
      .replace('{evidence_list}', evidenceList)
  }
}
```

## 七、业务裁决层设计（融合推理链路）

### 7.1 核心理念

**业务裁决层不是简单的"能不能做"，而是深度融合推理链路的"应该怎么做"**

```
┌─────────────────────────────────────────────────────────────────┐
│                   浅层裁决 vs 深度裁决                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  浅层裁决（当前错误设计）：                                      │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐                    │
│  │ 用户请求 │ -> │ 权限检查 │ -> │ 返回ability │                │
│  └─────────┘    └─────────┘    └─────────┘                    │
│                       ❌ 只回答能不能做，不回答怎么做            │
│                                                                  │
│  深度裁决（正确设计）：                                          │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐   │
│  │ 用户请求 │ -> │ 证据收集 │ -> │ 推理链路 │ -> │ 决策建议 │   │
│  └─────────┘    └─────────┘    └─────────┘    └─────────┘   │
│                       ✅ 回答做什么、怎么做、风险如何            │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 7.4 裁决器实现示例

```typescript
// ===== 优先级裁决器实现 =====

class PriorityVerdictResolver implements VerdictResolver {
  constructor(
    private llm: LLMClient,
    private evidenceCollector: EvidenceCollector,
    private economicCalculator: EconomicCalculator
  ) {}

  async resolvePriority(query: string, context: Context): Promise<VerdictResult> {
    // 1. 收集多维度证据
    const evidences = await this.collectPriorityEvidences(query, context)

    // 2. 构建裁决 Prompt（融合证据链）
    const prompt = this.buildVerdictPrompt(query, evidences)

    // 3. LLM 推理
    const reasoning = await this.llm.complete({
      prompt,
      temperature: 0.3,
      output_format: 'json'
    })

    // 4. 解析裁决结果
    const verdict = this.parseVerdict(reasoning)

    // 5. 计算置信度
    const confidence = this.calculateConfidence(evidences, verdict)

    return {
      type: 'priority_decision',
      query,
      conclusion: verdict.conclusion,
      evidence_chain: { evidences, ... },
      reasoning_path: verdict.steps,
      confidence,
      traces: verdict.traces
    }
  }

  private async collectPriorityEvidences(
    query: string,
    context: Context
  ): Promise<Evidence[]> {
    const [taskEvidences, economicEvidences, historyEvidences] = await Promise.all([
      // 任务证据：deadline、状态、依赖
      this.collectTaskEvidences(context.projectId),
      // 经济证据：延误成本、资源占用
      this.collectEconomicEvidences(context.projectId),
      // 历史证据：类似任务的完成时间
      this.collectHistoryEvidences(query)
    ])

    return [
      ...taskEvidences,  // E1, E2, E3
      ...economicEvidences,  // E4, E5
      ...historyEvidences  // E6, E7
    ]
  }

  private buildVerdictPrompt(query: string, evidences: Evidence[]): string {
    const evidenceList = evidences.map(e => `
[E${e.id}] ${e.source}: ${e.content}
  可靠性: ${e.reliability}
  关联性: ${e.relevance}
`).join('\n')

    return `
你是一个专业的优先级决策助手，需要基于以下证据进行深度推理。

【决策问题】
${query}

【证据链】
${evidenceList}

【输出要求】
请严格按照以下格式输出：

## 推理过程

### 步骤1：证据整合
- 各证据对问题的影响程度

### 步骤2：冲突检测
- 检查证据间是否有矛盾

### 步骤3：优先级计算
- 综合评分计算过程

## 裁决结果

### 结论
[应该优先做什么，清晰明确]

### 建议行动
1. [行动1]
2. [行动2]

### 风险提示
- 风险1（概率X%）：...
- 风险2（概率Y%）：...

### 置信度
- 最终置信度：XX%

### 推理痕迹
1. 根据[E1]...
2. 根据[E2]...
`
  }
}
```
