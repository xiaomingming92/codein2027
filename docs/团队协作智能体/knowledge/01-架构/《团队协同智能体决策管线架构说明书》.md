# 团队协同智能体 — 决策管线架构说明书

> 本文档阐述团队协同智能体的核心决策管线（Decision Pipeline），基于 LangGraph StateGraph 构建，遵循 **意图解析 → 证据检索 → 深度推理 → 交互点检测 → 业务裁决 → 回复生成** 的六节点流水线。

> **宣讲文档**：[《传统Agent vs 团队协同智能体 — 推理链路与运行时对比》](./《传统Agent%20vs%20团队协同智能体%20—%20推理链路与运行时对比》.md) — 一句话讲清传统 Agent 和本项目在推理链路与运行时上的本质差异

---

## 一、整体概览

### 1.1 管线的本质

该智能体不是简单的 RAG → Prompt → Response 模式，而是一条**多阶段可审计的决策管道**。每个阶段有独立的输入/输出边界，通过共享的 `AgentState` 串联，由 LangGraph 引擎调度执行。

### 1.2 设计约束

| 约束 | 说明 |
|------|------|
| RAG 必须参与 | 所有意图都走 retrieval，禁止 AI 随意发挥 |
| 推理与裁决分离 | reasoning 产出初步推理结论，verdict 在此基础上做二次裁决 |
| 交互点优先于裁决 | 检测到需要用户决策时，直接跳过 verdict 进入 response |
| 可审计性 | 每个节点执行都有进入/退出/错误的完整审计记录 |
| 全链路追踪 | traceId 贯穿所有节点，支持查询完整调用链 |

## 二、状态定义 (AgentState)

状态定义在 [src/agents/state.ts](file:///home/xmm/ai/农业智能体/team-coordinator-agent/src/agents/state.ts)，基于 `Annotation.Root` 构建。

### 2.1 核心状态字段

| 字段 | 类型 | 职责 | 写入节点 |
|------|------|------|---------|
| `messages` | `Message[]`（累加式 reducer） | 对话历史，新消息 append 而非覆盖 | response |
| `user` | `User \| null` | 当前用户信息 | 外部输入 |
| `project` | `Project \| null` | 当前项目上下文 | 外部输入 |
| `currentTask` | `CurrentTask \| null` | 当前任务快照（intent/entities/query）| intention |
| `evidenceChain` | `Evidence[] \| null` | 证据链数组 | retrieval |
| `verdictResult` | `Verdict \| null` | 推理/裁决结果 | reasoning, verdict |
| `retrievalContext` | `RetrievalContext \| null` | 检索聚合上下文 | retrieval |
| `explicitIntent` | `string \| null` | 外部显式意图（跳过 LLM） | 外部输入 |
| `structuredResponse` | `StructuredAgentResponse \| null` | 最终结构化响应体 | response |
| `conversationContext` | `Record<string, unknown>` | 会话上下文（含 traceId） | 贯穿 |
| `pendingInteraction` | `InteractionPoint \| null` | 待用户决策的交互点 | interactionPointDetection |

### 2.2 关键状态设计

**消息累加器**：`messages` 使用 `(x, y) => x.concat(y)` reducer，确保多轮对话历史自然累积而非覆盖。

**显式意图优先**：`explicitIntent` 允许外部直接指定意图，绕过 intention 节点的 LLM 解析，实现快速通道。

**traceId 贯穿**：通过 `conversationContext.traceId` 在节点间传递，用于全链路审计和追踪。

## 四、六节点详解

### 4.1 intention — 意图解析

**源文件**：[src/agents/nodes/intention.ts](file:///home/xmm/ai/农业智能体/team-coordinator-agent/src/agents/nodes/intention.ts)

**职责**：分析用户最后一条消息，提取意图类型和实体。

**执行流程**：

```
用户消息
  │
  ├─ 消息为空或非 user 角色 → 跳过，不更新 currentTask
  ├─ 消息有内容 →
  │     ├─ 多模态检测：识别 image_url / audio_url 类型
  │     ├─ 文本提取：拼接 text 片段为 queryText
  │     ├─ explicitIntent 存在？
  │     │   ├─ 是 → 快速通道：直接使用，跳过 LLM 调用
  │     │   └─ 否 → LLM 意图解析：
  │     │         ├─ 构建 IntentionInput → 调用 LLM
  │     │         ├─ 解析 JSON → 校验 intent 值域
  │     │         └─ 失败 → fallback 为 "chat"
  │     └─ 输出：currentTask { intent, entities, query }
```

**意图类型枚举**：`analysis` | `planning` | `question` | `decision` | `creation` | `modification` | `chat`

**显式意图快速通道**：当 `explicitIntent` 由外部直接指定时，intention 节点完全跳过 LLM 调用。这支持外部系统精确控制智能体行为，同时提高性能。

**Prompt 设计**（[intention.ts](file:///home/xmm/ai/农业智能体/team-coordinator-agent/src/agents/prompts/intention.ts)）：
- 支持纯文本和多模态两种模板
- 输出格式：`{ intent, entities: { projectId, taskId, keywords }, multimodal? }`
- parse 层做防御性清洗（去除 markdown 代码块标记）和值域校验

### 4.2 retrieval — 证据检索

**源文件**：[src/agents/nodes/retrieval.ts](file:///home/xmm/ai/农业智能体/team-coordinator-agent/src/agents/nodes/retrieval.ts)

**职责**：从多种来源聚合信息，构建完整证据链。

**执行流程**：

```
currentTask, project
  │
  ├─ 1. 证据链快照：基于现有 evidenceChain 创建副本
  ├─ 2. ChainTracer 记录：setEvidenceBefore() 记录检索前证据
  ├─ 3. 项目上下文注入：project 作为 context 类证据（reliability=0.9）
  ├─ 4. 知识库 RAG 检索：
  │     ├─ searchKnowledgeDocuments(query, 5) 向量检索
  │     ├─ 识别全文文档请求 → 调整 type 为 full_document
  │     ├─ 无结果 → 追加警告类证据（reliability=0.0）
  │     ├─ stream.evidenceFound() 实时推送证据发现
  │     └─ stream.ragResult() 推送检索汇总
  ├─ 5. 关键词证据：从 currentTask.entities.keywords 提取（reliability=0.7）
  ├─ 6. 多模态标记：图片/音频标记为证据（reliability=0.6）
  ├─ 7. 证据差异记录：tracer.recordEvidenceDiff()
  └─ 输出：evidenceChain[], retrievalContext, 结构化统计
```

**证据类型与可靠性**：

| 证据来源 | reliability | 说明 |
|---------|------------|------|
| 项目上下文 | 0.9 | 高可靠性，来自数据库 |
| 知识库文档（RAG 命中）| 0.7~0.9 | 根据相似度评分 |
| 关键词匹配 | 0.7 | 间接证据 |
| 多模态输入 | 0.6 | 非结构化，需要额外解读 |
| 无检索结果警告 | 0.0 | 明确标记为不可靠 |

**结构化输出**：检索完成后计算总得分（∑ reliability × relevance 加权和）和来源分布，通过 SSE 流式推送。

### 4.3 reasoning — 深度推理

**源文件**：[src/agents/nodes/reasoning.ts](file:///home/xmm/ai/农业智能体/team-coordinator-agent/src/agents/nodes/reasoning.ts)

**职责**：基于证据链进行多步推理，产出推理路径、结论和置信度。这是管线中的**初步推理层**。

**执行流程**：

```
evidenceChain, currentTask.query
  │
  ├─ 空证据快速返回：
  │     无 query 或无证据 → final_confidence=0 的空裁决结果
  │
  ├─ LLM 推理：
  │     输入：证据列表（id/source/content/reliability）
  │     输出：reasoning_path / traces / conclusion / confidence
  │
  ├─ 解析校验：
  │     ├─ 成功 → 提取 reasoning_path（步骤序列）、traces、conclusion、confidence
  │     └─ 失败 → confidence=0 的空结果（不中断管线）
  │
  └─ 输出：verdictResult { reasoning_path, conclusion, confidence }
```

**置信度四元组**：

| 字段 | 说明 |
|------|------|
| `base_confidence` | 基础置信度 |
| `reliability_discount` | 证据可靠性折扣 |
| `conflict_discount` | 冲突/风险折扣 |
| `final_confidence` | 最终置信度 = base - reliability_discount - conflict_discount |

**Prompt 设计**（[reasoning.ts](file:///home/xmm/ai/农业智能体/team-coordinator-agent/src/agents/prompts/reasoning.ts)）：
- 输入：query + 证据列表
- 输出：`{ reasoning_path, traces, conclusion, confidence }`
- reasoning_path 为步骤序列，每步描述推理动作和引用证据

### 4.4 interactionPointDetection — 交互点检测

**源文件**：[src/agents/nodes/interaction-point-detection.ts](file:///home/xmm/ai/农业智能体/team-coordinator-agent/src/agents/nodes/interaction-point-detection.ts)
**Prompt 定义**：[src/agents/prompts/interaction-point-detection.ts](file:///home/xmm/ai/农业智能体/team-coordinator-agent/src/agents/prompts/interaction-point-detection.ts)
**类型定义**：[src/agents/types/structured-output.ts](file:///home/xmm/ai/农业智能体/team-coordinator-agent/src/agents/types/structured-output.ts#L71-L80)

**职责**：在分析和规划类意图中检测是否需要用户介入做决策。这是管线的**关键分叉点**，也是 **Human-in-the-Loop** 的关键入口。

#### 4.4.1 为什么需要交互点检测？

reasoning 节点产出推理结论后，有些结论是清晰的、可以直接进入 verdict 裁决，但有些结论中包含了**需要人类做判断的关键抉择**——比如"方案 A 成本低但周期长，方案 B 成本高但周期短"，机器无法替代人类决定"成本"和"时间"哪个更重要。交互点检测的目的就是**在推理完成之后、裁决之前**，识别出这些需要人类介入的时刻，把决策权交还给用户。

#### 4.4.2 触发条件

交互点检测**仅在以下两种意图下激活**：

| 意图 | 说明 | 典型场景 |
|------|------|---------|
| `analysis` | 分析类 | 项目分析、风险评估、数据解读 |
| `planning` | 规划类 | 任务规划、资源排期、路径设计 |

其他意图（`question` / `decision` / `creation` / `modification` / `chat`）直接跳过，`pendingInteraction` 返回 `null`。

此外，如果 reasoning 节点未产出 `verdictResult.conclusion`（无结论），也直接跳过。

#### 4.4.3 四种交互点类型

LLM 会将推理结论与证据链对比，判断是否存在以下四类交互点：

| 类型 | 含义 | 典型触发条件 | 示例问题 |
|------|------|-------------|---------|
| `decision` | **决策点** — 需要用户在多个方案中做出选择 | 推理结论包含多个可行方案，各有优劣，无法自动判定最优 | "根据分析，项目有两种推进路径：方案A（先做技术验证再全面铺开）或方案B（边开发边验证），你需要决策选择哪种？" |
| `tradeoff` | **权衡点** — 需要用户在互斥维度之间做出取舍 | 推理结论揭示两个维度存在互斥关系（如成本vs质量、速度vs风险） | "当前资源只能支持一个方向：优先完成高优任务（但会延迟其他）还是平衡推进（但整体进度较慢）？" |
| `clarification` | **澄清点** — 信息不足，需要用户补充关键信息 | 证据链中存在信息缺口，或推理结论依赖未确认的假设 | "推理结论依赖'项目 deadline 为 6 月底'这一前提，请确认这个时间是否准确？" |
| `confirmation` | **确认点** — 推理结论涉及高风险，需要用户确认 | 推理结论的置信度低于阈值，或建议行动涉及高风险操作 | "根据分析，建议立即切换技术方案，但这会导致已开发模块返工。是否确认执行此决策？" |

#### 4.4.4 InteractionPoint 数据结构

```typescript
interface InteractionPoint {
  type: "decision" | "tradeoff" | "clarification" | "confirmation"
  dimension?: string            // 决策维度，如 "cost_vs_time" / "priority" / "resource_allocation"
  description: string           // 向用户提出的具体问题描述
  options: Array<{
    label: string               // 选项展示标签，如 "方案A: 先做技术验证"
    reason: string              // 推荐理由，解释为什么这个选项值得考虑
    impact: string              // 选择该选项后的预期影响
  }>
}
```

**每个字段的含义：**

- **`type`** — 决定前端如何渲染交互界面。`decision` 和 `tradeoff` 通常渲染为多选卡片，`clarification` 渲染为输入提示，`confirmation` 渲染为确认对话框。
- **`dimension`** — 标记决策所在的业务维度，前端组件用它来决定展示哪些辅助信息（如 `cost_vs_time` 会触发成本对比图表）。
- **`description`** — 面向用户的问题描述，LLM 用自然语言解释当前面临的选择困境，让用户理解为什么需要做出决定。
- **`options`** — 可选方案列表，每个方案包含三个要素：
  - `label`：给用户看的选项名称
  - `reason`：为什么这个选项值得考虑（基于证据链推导）
  - `impact`：选择后的预期后果（帮助用户评估风险）

#### 4.4.5 LLM Prompt 结构

交互点检测的 Prompt 向 LLM 输入以下信息：

| 输入字段 | 来源 | 说明 |
|---------|------|------|
| 用户原始问题 (`query`) | `currentTask.query` | 用户的原始提问文本 |
| 证据链摘要 | `evidenceChain` 裁剪到前 200 字符 | 让 LLM 知道推理依据了什么数据 |
| 推理结论 | `verdictResult.conclusion.content` | reasoning 节点产出的结论文本 |
| 建议行动 | `verdictResult.conclusion.actions` | reasoning 节点产出的行动建议 |

LLM 被要求返回 JSON：`{ hasInteractionPoint: boolean, interactionPoint?: { type, dimension, description, options } }`

**parse 层的防御性处理**：
- 去除 markdown 代码块标记（`` ```json `` / `` ``` ``）
- `type` 值域校验：只接受四种合法类型，非法值 fallback 为 `"clarification"`
- `description` 保底为空字符串
- `options` 保底为空数组

#### 4.4.6 执行流程图

```
intent, verdictResult, evidenceChain
  │
  ├─ 意图过滤：
  │     非 analysis/planning → 直接返回 pendingInteraction: null
  │     审计: "非分析/规划意图, 跳过检测"
  │
  ├─ 结论前置检查：
  │     无 verdictResult.conclusion → 直接返回 pendingInteraction: null
  │     审计: "无推理结论, 跳过检测"
  │
  ├─ LLM 交互点检测：
  │     输入：query + 证据链摘要 + 推理结论 + 建议行动
  │     LLM 判断：是否存在需要用户做决策/权衡/澄清/确认的关键点
  │     ├─ LLM 调用失败 → pendingInteraction: null（防御性 fallback）
  │     └─ LLM 返回成功 →
  │           ├─ hasInteractionPoint = false → pendingInteraction: null
  │           │     审计: "未检测到交互点"
  │           └─ hasInteractionPoint = true →
  │                 ├─ 构造 InteractionPoint { type, dimension, description, options }
  │                 ├─ 审计: "检测到交互点 type=xxx, dimension=xxx, optionsCount=N"
  │                 ├─ SSE 推送: structured_update { interactionPoint }
  │                 └─ 输出: pendingInteraction = interactionPoint
  │
  └─ 最终输出：pendingInteraction | null
```

#### 4.4.7 对下游的影响

```
interactionPointDetection
  │
  ├─ pendingInteraction != null（有交互点）
  │     │
  │     ├─ routeByInteractionPoint → "response"（跳过 verdict）
  │     │     原因：用户需要先做出选择，裁决才有意义
  │     │
  │     └─ response 节点接收到 interactionPoint
  │           ├─ 路径 A：生成带有选项按钮的交互式回复
  │           └─ structuredResponse.interactionPoint 携带完整选项数据
  │               前端渲染为可点击/可输入的交互组件（按钮组/输入框/确认弹窗）
  │               用户选择后 → 下一条消息携带 explicitIntent=用户选择 → 重新走管线
  │
  └─ pendingInteraction == null（无交互点）
        └─ routeByInteractionPoint → "verdict"（继续裁决）
```

#### 4.4.8 具体示例

**输入场景**：用户问"我们项目下个月要交付，但人手不够，怎么办？"

### 4.5 verdict — 业务裁决

**源文件**：[src/agents/nodes/verdict.ts](file:///home/xmm/ai/农业智能体/team-coordinator-agent/src/agents/nodes/verdict.ts)

**职责**：基于证据链进行深度裁决，支持模型参数动态切换。这是管线中的**二次决策层**。

**执行流程**：

```
evidenceChain, currentTask.query
  │
  ├─ 1. 权重解析：calculateDefaultWeights() 均分权重
  │
  ├─ 2. LLM 裁决：
  │     输入：query + evidenceList + weights
  │     输出：裁决类型 + conclusion + confidence
  │     ├─ 成功 → 解析裁决结果
  │     └─ 失败 → PATH_SELECTION + confidence=0
  │
  └─ 输出：verdictResult { type, conclusion, confidence, traces }
```

**裁决类型枚举**：`PRIORITY_DECISION` | `RISK_ASSESSMENT` | `RESOURCE_ALLOCATION` | `COST_BENEFIT` | `TIMELINE_ESTIMATION` | `PATH_SELECTION`

### 6.2 StreamBus

源文件：[src/agents/stream-bus.ts](file:///home/xmm/ai/农业智能体/team-coordinator-agent/src/agents/stream-bus.ts)

traceId 级别的全局事件总线，实现 SSE 推送的注册/注销/发射机制：

```typescript

### 6.3 事件时序

```
route.ts: registerStreamBus(traceId, callback)

  ┌─ intention ────────────────────────────────────┐
  │  invoke(prompt) → parse →                      │
  │  structured_update { intent }                   │
  │  return { currentTask }                        │
  └────────────────────────────────────────────────┘

  ┌─ retrieval ────────────────────────────────────┐
  │  rag_search { query }                          │
  │  results = searchDocuments() → for each:       │
  │    evidence_found { evidence }  ← 逐条实时推送  │
  │  rag_result { count, sources }                 │
  │  structured_update { evidenceChain }           │
  │  return { evidenceChain }                      │
  └────────────────────────────────────────────────┘

  ┌─ reasoning ────────────────────────────────────┐
  │  invoke(prompt) → parse →                      │
  │  structured_update { reasoningPath }           │
  │  return { verdictResult }                      │
  └────────────────────────────────────────────────┘

  ┌─ interactionPointDetection ────────────────────┐
  │  invoke(prompt) → parse →                      │
  │  structured_update { interactionPoint }         │
  │  return { pendingInteraction }                  │
  └────────────────────────────────────────────────┘

  ┌─ verdict ──────────────────────────────────────┐
  │  invoke(prompt) → parse →                      │
  │  structured_update { verdict }                  │
  │  return { verdictResult }                      │
  └────────────────────────────────────────────────┘

  ┌─ response ─────────────────────────────────────┐
  │  response_start                                │
  │  stream = getLLM().stream(prompt)  ← 真流式！   │
  │  for await: token { content }   ← 逐 token     │
  │  response_end { fullContent }                  │
  │  structured_update { displayContent }           │

## 九、架构设计要点

### 9.1 双阶段裁决

reasoning 产出初步推理结论和置信度，verdict 在此基础上做二次裁决。推理与决策分离，使每个阶段可独立调试和替换。

### 9.2 交互点提前返回

当 interactionPointDetection 检测到需要用户决策时，直接跳过 verdict 进入 response。避免在无需裁决的场景下浪费算力，同时让用户及时介入决策。

### 9.3 三层审计 + 全链路追踪

- **Layer 1 开发审计**：`agentAuditNodeStart/End/Error` — 控制台 + 文件 + DB metadata
- **Layer 2 运行时审计**：`auditNodeEvent` — AuditLog 表，index traceId
- **Layer 3 调试日志**：`console.error`
- **ChainTracer**：记录 inputSnapshot / result / evidenceDiff / routeDecision

### 9.5 SSE 流控制与 Graph 解耦

`NodeStreamController` 在各节点内部推送事件到 `StreamBus`，前端通过 `registerStreamBus(traceId, push)` 订阅指定 traceId 的实时事件流。LLM token 逐字推送，结构化数据完整推送，流控制与 LangGraph 执行引擎互不依赖。

### 9.6 防御性编程

每个 LLM 调用的 parse 都有 try-catch 和 fallback 值域校验，失败路径的信息密度与成功路径等价（ADD-6 合规），任何节点异常都不会中断整条管线。
